"The Y combinator is the lambda calculus equivalent of a strange loop — a structure that refers to itself without ever naming itself."
— Douglas Hofstadter, Gödel, Escher, Bach

You might have heard of heavy CS terms like "self-referential code", Quines, or even the (original) Y Combinator. With these problems, you will try to implement some of these in bash!

In 1936, before computers existed, Alonzo Church invented the lambda calculus — a formal system where the only primitive is the anonymous function. No numbers, loops or named procedures. Just functions applied to arguments.

The surprising result: that is enough. Everything computable can be expressed this way.

Pure lambda calculus has no assignment, so you cannot write:

factorial = λn. if n=0 then 1 else n * factorial(n-1)

factorial does not exist yet on the right-hand side when you write the definition. You cannot name a function and call it from within its own body.

So how do you get recursion? You use a fixed-point combinator.

A fixed point of a function f is a value x such that f(x) = x. The Y combinator (the original), discovered by Haskell Curry, constructs the fixed point of any function: 

Y = λf. (λx. f(x x))(λx. f(x x))

A primer: In lambda calculus land, arguments "applied" to a function are just written next to them so a + b becomes + a b, and multi-argument functions are implemented using currying — a 2-argument function is just a function that returns a function that returns a number, so + = λa. (λb (...)). (Implementing addition in lambda calculus is a very different task but hopefully this clears things.)

Now, unfold it once: Y(f) = (λx. f(x x))(λx. f(x x)) = f((λx. f(x x))(λx. f(x x))) = f(Y(f)).

So Y(f) = f(Y(f)) — the fixed point equation holds. The Y combinator hands f a self-reference and f uses it to recur, without ever naming itself.

In Python (using the strict-evaluation-safe Z combinator):

Z = lambda f: (lambda x: f(lambda *v: x(x)(*v)))(
               lambda x: f(lambda *v: x(x)(*v)))

We will deal with this later. Let's get started.

Problem 2 Don't Quine Already!

A quine is a program that outputs its own source code exactly — no arguments, no file reads, no network access. Pure self-reproduction.

Formally: if run(P) is the stdout of program P, a quine Q satisfies:

run(Q) = Q

This is a fixed point of run. Kleene's Second Recursion Theorem guarantees that such a fixed point exists in any Turing-complete language.

Part A — Cheating the Game

We present quine.sh such that bash quine.sh outputs the exact contents of quine.sh. Let's see if you can guess this 2-liner script.

To verify:

bash quine.sh > out.txt
diff out.txt quine.sh && echo "QUINE VERIFIED"

The solution is…

+ reveal solution — quine.sh 
#!/bin/bash
cat "$0"

$0 is the path to the currently executing script. Clean — but a theorist would say this is not a "true" quine since it reads source from disk rather than reconstructing it from internal data alone.

Part B — We're Purists

Write quine_pure.sh using only printf and a string variable s. No cat, no $0, no file access.

The structure:

#!/bin/bash
s=<source encoded as a string, containing a %s placeholder>
printf "$s" "$s"

When printf "$s" "$s" runs, %s is replaced by the value of $s, which reconstructs the s=... line. The key challenge: the string s must contain a representation of itself, including the surrounding quote characters.

note This is supposed to be an open-ended challenge. You can look at standard quines in other languages but that would give it away. You are encouraged to try this out yourself first — don't quine already!
hint The quote character ' has ASCII code 39 (octal \047). Inside a $'...' literal in bash, you can write \047 to get a literal single quote without closing the string. This lets you encode the s='...' line inside s itself.

Problem 3 The Classier, More Chic Combinator

Here is a classical implementation of a variant of the fixed-point operator in Python. Note the peculiar syntax, so as to avoid undefined behaviour. 

Z = lambda f: (lambda x: f(lambda *v: x(x)(*v)))(
               lambda x: f(lambda *v: x(x)(*v)))

fact_template = lambda f: lambda n: 1 if n <= 1 else n * f(n - 1)
factorial = Z(fact_template)
print(factorial(5))  # 120

The key: fact_template is a true lambda function so it cannot "name" itself. Hence, it receives f (usually just itself) as a self-reference and never names itself. Z implements the Z combinator. When passed fact_template, it produces a lambda expression with the value of f substituted.

Notice that here v has been added to ensure that the final lambda created (factorial) can accept arguments and lazily evaluate the expression recursively.

Here Z(fact_template) returns f(h) where h = lambda *v: g(g)(*v) and g = lambda x: f(lambda *v: x(x)(*v)) and f = fact_template, which is in turn just a function.

Observe what happens when factorial(5) is called. It first evaluates f(h)(5) which reduces to 5 * h(4). The second term reduces to g(g)(4) but notice! Z(fact_template) is in fact simply a lambda of the form lambda x: g(g)(x)… we have recursion!! This reduces to 4 * h(3) and so on until we hit a base case.

Think about what happens if you call the Z combinator on a recursively defined function which does not have a base case… Can you always determine if a given function satisfies this property? Shout out to Gödel.

In bash, we cannot pass functions as arguments. But we can pass script paths. Executing a script with arguments is function application. Passing "$0" — the script's own path — is self-application.

You are given fact_templ.sh — the factorial template. This is the bash equivalent of fact_template in the Python code above. It receives a combinator as $1 and uses it to recurse — never naming itself.

#!/usr/bin/env bash
combinator="$1"
n="$2"

if [ "$n" -le 1 ]; then
    echo 1
else
    # Recursive call: ask the combinator to apply the template again with n-1
    prev=$( "$combinator" "$0" $(( n - 1 )) )
    echo $(( n * prev ))
fi

Problem 4 Let's Interpret

Enough self-modification and hate against the interpreter. We will now build an interpreter for linear expressions to gain appreciation for our foe.

This is meant for your practice and for gaining comfort with writing heavier bash code; the later parts involve really advanced code so don't be disheartened if it takes a lot of effort to go through those.

Inputs

funcs.txt

One function per line. Each function takes two parameters and has a right-hand side expression:

f a b 2*a+b
g a b a+2*b
h a b f(2*a+b,a)
i a b g(a+b,2*b)

The RHS is one of exactly two things:

  • A linear expression in a and b: things like 2*a+b, a+2*b, 3*a, a+b. No spaces, no parentheses. Coefficients are plain integers. The expression variables are always literally a and b (the parameter names in the file are just for readability).
  • A composition: a call to a previously defined function with two linear expressions as arguments: f(2*a+b,a), g(a+b,2*b). The callee must appear earlier in the file. No nesting beyond one level here.

calls.txt

Expressions in prefix notation, one per line:

f 2 3
f 1 g 3 4
h g 2 3 f 1 4
i h 2 3 g 1 2

In prefix notation the function always comes before its two arguments, which can themselves be calls. So f 1 g 3 4 means f(1, g(3,4)), and h g 2 3 f 1 4 means h(g(2,3), f(1,4)). Work out i h 2 3 g 1 2 yourself before running.

Part C — Prefix Call Evaluator

Now handle calls.txt. The grammar for a value in prefix notation is:

val ::= integer
      | funcname val val

So to evaluate a line you need to walk the token stream and consume tokens as you go — a token consumed by a recursive call to parse_val must not be seen again by the caller. This means you need a global index into the token array that all recursive calls share. 

TOKENS=()
IDX=0

parse_val() {
    local tok="${TOKENS[$IDX]}"
    IDX=$(( IDX + 1 ))
    if [[ $tok =~ ^-?[0-9]+$ ]]; then
        PARSE_RESULT=$tok                        # literal: we're done
    else
        parse_val; local _a=$PARSE_RESULT        # consume first argument
        parse_val; local _b=$PARSE_RESULT        # consume second argument
        PARSE_RESULT=$("$tok" "$_a" "$_b")       # call the function
    fi
}

For each line in calls.txt, split it into TOKENS, reset IDX=0, call parse_val, and print:

f 2 3            =>  7
g 4 5            =>  14
f 1 g 3 4        =>  13
h g 2 3 f 1 4    =>  52
i h 2 3 g 1 2    =>  41
questions
  • Why must IDX and PARSE_RESULT be globals? Try making IDX local and see what breaks. What's the bash-specific reason this fails, relating to how function calls work in bash vs other languages?
  • What does your parser do when a line in calls.txt has too few tokens? Too many?

interp.sh

#!/bin/bash

# Part A: evaluate a linear expression in a,b given concrete values
eval_linear() {
    local expr=$1 a=$2 b=$3
    local s="${expr//a/$a}"
    s="${s//b/$b}"
    echo $(( s ))
}

# Part B: generate and register bash functions from funcs.txt
build_func() {
    local name=$1 p1=$2 p2=$3 rhs=$4
    local fn_pat='^([a-z])\((.+),(.+)\)$'
    if [[ $rhs =~ $fn_pat ]]; then
        # composition RHS: callee(expr1, expr2)
        local callee="${BASH_REMATCH[1]}"
        local expr1="${BASH_REMATCH[2]}" expr2="${BASH_REMATCH[3]}"
        cat << FUNCDEF
${name}() {
    local ${p1}=\$1 ${p2}=\$2
    local _a1=\$(eval_linear "${expr1}" "\$${p1}" "\$${p2}")
    local _a2=\$(eval_linear "${expr2}" "\$${p1}" "\$${p2}")
    ${callee} "\$_a1" "\$_a2"
}
FUNCDEF
    else
        # linear RHS
        cat << FUNCDEF
${name}() {
    local ${p1}=\$1 ${p2}=\$2
    eval_linear "${rhs}" "\$${p1}" "\$${p2}"
}
FUNCDEF
    fi
}

while IFS=' ' read -r name p1 p2 rhs; do
    eval "$(build_func "$name" "$p1" "$p2" "$rhs")"
done < funcs.txt

# Part C: recursive descent prefix evaluator
TOKENS=()
IDX=0

parse_val() {
    if (( IDX >= ${#TOKENS[@]} )); then
        echo "Error: unexpected end of input" >&2
        exit 1
    fi

    local tok="${TOKENS[$IDX]}"
    IDX=$(( IDX + 1 ))

    if [[ $tok =~ ^-?[0-9]+$ ]]; then
        PARSE_RESULT=$tok
    else
        if ! declare -f "$tok" > /dev/null; then
            echo "Error: unknown function '$tok'" >&2
            exit 1
        fi

        parse_val; local _a=$PARSE_RESULT
        parse_val; local _b=$PARSE_RESULT
        PARSE_RESULT=$("$tok" "$_a" "$_b")
    fi
}

while IFS=' ' read -r line; do
    read -ra TOKENS <<< "$line"
    IDX=0
    parse_val
    echo "$line  =>  $PARSE_RESULT"
done < calls.txt
